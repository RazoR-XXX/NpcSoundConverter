# Форматы файлов озвучки NPC в Rust (реверс-инжиниринг)

> Автор: **RazoR**, 2026. English version: **[NPC_Sound_Formats.md](NPC_Sound_Formats.md)**.
> Discord: <https://discordapp.com/users/1056019567589216336>
>
> Полная спецификация `.json`-файлов озвучки для плагинов NPC на Rust/Oxide/Carbon:
> - **HumanNPC** (*Razor*) — пакеты Steam Voice
> - **XDQuest** (*DezLife*) — Ogg Vorbis в base64
>
> Документ написан потому, что **публичной спецификации этих форматов не было**, а все
> «туториалы» сводятся к «запиши в игре через микрофон». Ниже — полный разбор по байтам,
> чтобы любой мог собрать файл напрямую из аудио.

---

## Быстрое сравнение

| | **HumanNPC** (Razor) | **XDQuest** (DezLife) |
|---|---|---|
| Ключ JSON | `"Data"` | `"voiceData"` + `"audioType"` + `"durationSeconds"` |
| Содержимое | GZip → пакеты Steam Voice (Opus) | сырой **Ogg Vorbis** (base64) |
| Параметры аудио | Opus Hybrid SWB, моно, 24 кГц | Ogg Vorbis, моно, 48 кГц |
| Как проигрывается | сервер шлёт пакеты → клиент `DecompressVoice` | плагин декодирует Ogg → шлёт как голос |
| Сложность | очень высокая (CRC32, seq, упаковка) | простая (просто ogg-файл в base64) |
| Папка | `data/HumanNPC/Sounds/` | `data/XDQuest/Sounds/` |

# Часть 1 — Формат XDQuest (простой)

XDQuest хранит озвучку как обычный Ogg Vorbis файл, закодированный в base64:

```json
{
  "voiceData": "<base64 Ogg Vorbis файла>",
  "audioType": 0,
  "durationSeconds": 4.53
}
```

| Поле | Значение |
|------|----------|
| `voiceData` | Ogg Vorbis файл в base64 |
| `audioType` | всегда `0` |
| `durationSeconds` | длительность в секундах (float) |

Ogg внутри: **моно, 48000 Гц, кодек Vorbis** (качество ~5, ~128 кбит/с VBR).

Генерация:
```bash
ffmpeg -i input.mp3 -vn -ac 1 -ar 48000 -c:a libvorbis -q:a 5 output.ogg
```
Затем `base64(output.ogg)` → `voiceData`, длительность из ffprobe, собрать JSON. Никакой
спец-упаковки, CRC и Opus — предельно просто.

# Часть 2 — Формат HumanNPC (сложный)

## Кратко

Файл озвучки HumanNPC — это **НЕ аудиофайл**. Это поток пакетов **Steam Voice** — те же
байты, что делает клиент Rust через `SteamUser.GetVoice(...)` и принимает через
`SteamUser.DecompressVoice(...)`.

```
sound.json  =  { "Data": "<base64>" }
<base64>    =  GZip( voiceChunkBlob )
voiceChunkBlob = [int32 LE len0][int32 LE len1]...[int32 LE lenN]   (по одному на чанк)
                 ++ chunk0 ++ chunk1 ++ ... ++ chunkN               (байты чанков)
каждый чанк = один пакет Steam Voice (см. ниже)
```

Самая важная и легко упускаемая деталь: внутри каждого чанка каждый Opus-фрейм обёрнут в
заголовок `[u16 длина][u16 порядковый_номер]`, и фреймы должны быть **одиночными 20 мс
(TOC code 0)**, а НЕ одним большим многофреймовым пакетом (code 3).

## Слой 1 — JSON

Плагин сериализует класс с одним свойством через кастомный конвертер. На диске:
```json
{ "Data": "H4sIAAAAAAA...base64..." }
```
base64 декодируется в **GZip** (магия `1F 8B`). Внутри — список голосовых чанков.

## Слой 2 — GZip + контейнер чанков

```
[int32 LE: длина chunk0]
[int32 LE: длина chunk1]
...
[int32 LE: длина chunkN]
байты chunk0
байты chunk1
...
```
То есть **сначала все длины**, потом **все байты чанков подряд**. Весь блоб сжат GZip и
закодирован в base64 в поле `"Data"`.

Как плагин **проигрывает** чанк: шлёт каждый чанк ближайшим клиентам как сетевое голосовое
сообщение `VoiceData` (с EntityID = сетевой ID NPC), по одному каждые ~0.07 с. Клиент
передаёт байты в Steam-декодер голоса (`DecompressVoice`).

## Слой 3 — чанк = пакет Steam Voice

```
+----------------------------------------------------------+
| 8 байт   | SteamID64 «говорящего» (uint64, little-endian) |
+----------------------------------------------------------+
| payload'ы (один или несколько, см. ниже)                  |
+----------------------------------------------------------+
| 4 байта  | CRC32 (IEEE) всего пакета выше (LE)             |
+----------------------------------------------------------+
```

### Payload'ы

Каждый payload: 1 байт типа, затем `u16 LE` значение:

| Тип | Имя | Значение `u16` | Далее |
|-----|-----|----------------|-------|
| `0`  | Тишина (Silence) | число «тихих» сэмплов | (ничего) |
| `6`  | OPUS PLC | длина данных в байтах | столько байт **Opus PLC данных** (слой 4) |
| `11` | Sample Rate | частота (Гц) | (ничего) |

Обычный чанк = payload `SampleRate(11)`, затем один/несколько `OpusPlc(6)`. Частота —
**24000 Гц** (оптимальная для Steam Voice), кодируется как `0B C0 5D` (`0x0B` = тип,
`0x5DC0` = 24000).

### CRC32

- Стандартный **CRC32 / IEEE** (полином `0xEDB88320`, init `0xFFFFFFFF`, финальный XOR `0xFFFFFFFF`).
- Считается по всему пакету **кроме последних 4 байт**.
- Хранится little-endian в последних 4 байтах.
- **Клиент отбрасывает пакет с неверной CRC** — поэтому «валидный Opus» без CRC играет как
  полная тишина.

## Слой 4 — внутри OPUS PLC payload

Байты внутри payload типа `6` — это **не один Opus-пакет**, а последовательность фреймов с
префиксами длины и порядкового номера:

```
повторять пока payload не кончится:
    [u16 LE: длина фрейма]
    если длина == 0xFFFF:   // маркер сброса: сбросить состояние декодера, seq = 0, continue
        continue
    [u16 LE: порядковый номер (seq)]
    [длина_фрейма байт: один Opus-фрейм]
```

- `seq` — **глобальный, монотонно растущий** счётчик по всему потоку (для concealment
  потерь): `seq < ожидаемого` → сброс декодера; `seq > ожидаемого` → PLC (восстановление
  потерянных фреймов).
- `0xFFFF` — спец-маркер сброса (без seq/данных).

### Сами Opus-фреймы

Рабочие файлы (запись в игре) используют:

| Свойство | Значение |
|----------|----------|
| Режим Opus | **Hybrid**, SuperWideBand (TOC config 13) |
| Длительность | **20 мс** (480 сэмплов @ 24 кГц) |
| TOC code | **0** (один фрейм на Opus-пакет) |
| Каналы | моно |
| Размер фрейма | ~60–100 байт |

Для гладкого воспроизведения нужен **SILK-режим** (SILK или Hybrid) — он для речи и имеет
чистые стыки фреймов:
- ✅ **Hybrid SuperWideBand (config 13), 20 мс, code 0** — то, что делает игровая запись, и
  единственный режим, гладкий на реальном клиенте. Нужен полноценный кодировщик Opus
  (**нативный `libopus` / opus.dll**): `OPUS_APPLICATION_VOIP` + `SET_BANDWIDTH(SUPERWIDEBAND)`
  + `SET_SIGNAL(VOICE)` + **DTX**.
- ⚠️ SILK-only WideBand (config 9) — декодируется и оффлайн выглядит гладко, но на реальном
  клиенте даёт **щелчки на границах пакетов**.
- ⚠️ CELT-only (config 27) — играет, но из-за pre-echo CELT даёт **периодический затык** на речи.

Что **не работает совсем**:
- ❌ Один большой многофреймовый пакет (40/60 мс, TOC **code 3**).
- ❌ Сырой Opus без обёртки `[len][seq]`.
- ❌ Неверная/отсутствующая CRC32.
- ❌ Обычный Ogg (для HumanNPC; это формат XDQuest).

### Упаковка пакетов важна для ПОВТОРНОГО воспроизведения (критично!)

Даже при идеальном аудио **раскладка пакетов** влияет на повтор озвучки одного и того же
NPC. С новым (только что созданным) NPC первый проигрыш всегда чистый; последующие глючат на
фиксированном месте, если поток не упакован как игровая запись. Нужно так:

- **Первый пакет (чанк): одна группа ~8 фреймов (~160 мс)** — заполняет джиттер-буфер клиента.
- **Последующие чанки: ДВЕ группы `SampleRate + OpusPlc`**, по ~3 фрейма (≈6 фреймов / 120 мс
  на чанк). То есть в каждом пакете частота объявляется перед каждой мелкой группой.
- **Конец потока: маркер сброса `0xFFFF`** в конце последней opus-группы, затем финальный
  пакет только с payload'ом **`Silence` (тип 0, ~1500 сэмплов / 62 мс)**.
- Включить **DTX** — паузы кодируются 1-байтовыми фреймами (как у записи).

Равномерная упаковка (например, по 5 фреймов в одной группе на чанк) декодируется в
**идентичный PCM оффлайн**, но даёт воспроизводимый затык на 2-м и последующих проигрышах в
игре. Групповая раскладка выше играет чисто на каждом повторе.

## Как собрать файл из произвольного аудио

1. **Декодировать** вход в PCM: **моно, 24000 Гц, signed 16-bit LE**
   (`ffmpeg -i in.mp3 -vn -ac 1 -ar 24000 -f s16le -acodec pcm_s16le out.pcm`).
2. **Кодировать** PCM в Opus **фреймами по 20 мс** (480 сэмплов), моно, **Hybrid SWB** через
   нативный libopus.
3. **Обернуть** каждый фрейм в `[u16 len][u16 seq]`; `seq` растёт глобально.
4. **Сгруппировать** в пакеты по раскладке выше, добавить `[0x0B][u16 24000]` (частота) и
   `[0x06][u16 длина]`, спереди 8-байтный SteamID64, в конце CRC32.
   Завершить сбросом `0xFFFF` + финальной тишиной.
5. **Сериализовать**: блоб «сначала длины» → GZip → base64 → `{"Data":"..."}`.
6. Сохранить в `.../oxide|carbon/data/HumanNPC/Sounds/<имя>.json`.

В игре: `/npc_edit` → `/npc sound <имя>` → `/npc soundonuse true` → `/npc_end`
(`o.reload HumanNPC` / `carbon.reload HumanNPC` — сбросить кэш звуков плагина).

## Частые причины «тишины»/проблем

| Симптом | Причина |
|---------|---------|
| Полная тишина, файл «выглядит ок» | Нет/неверная CRC32 → клиент отбрасывает каждый пакет |
| Полная тишина | Сырой Opus без обёртки `[u16 len][u16 seq]` |
| Полная тишина | Многофреймовые пакеты (TOC code 3) вместо code 0 |
| Файл никто не читает | Это Ogg / `{"voiceData":...}` — формат XDQuest, не HumanNPC |
| Лёгкий периодический затык | CELT на речи — используйте Hybrid SWB (нативный libopus) |
| Щелчки на границах | SILK-only — используйте Hybrid SWB |
| 1-й раз чисто, повторы с затыком | Равномерные пакеты в 1 группу — нужна групповая раскладка |

## Источники

- Формат подтверждён по декомпиляции `Facepunch.Steamworks` (`SteamUser.GetVoice` /
  `DecompressVoice`, оптимальная частота 24000) и исходникам плагина HumanNPC.
- Steam Voice сверен с **demostf/steam-audio-codec** и блогом **«Reversing Steam Voice
  Codec»** (*Zhenyang Li*).
- Opus: RFC 6716. Кодировщик в референсном конвертере: нативный **libopus**.

*Материалы внешних источников переработаны с учётом лицензионных ограничений.*
